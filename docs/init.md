# Init Command

`architect-companion init` bootstraps `.architect-companion/` in a consuming
repository from selected profiles and renders enabled targets. It is the entry
point most teams use first.

See [Harness Model](harness-model.md) for the broader CLI surface and
[Profile Model](profile-model.md) for what profiles contribute.

## Contract

A single invocation of `init` is atomic:

1. **Pre-flight.** Validate inputs, confirm `.architect-companion/` does not
   already exist, and ensure would-be render-target paths are missing or
   already generated by Architect Companion.
2. **Scaffold.** Write `.architect-companion/harness.yml` and
   `.architect-companion/architecture/boundaries.yml`.
3. **Lock.** Compute and write `.architect-companion/profile.lock.yml`.
4. **Render.** Invoke the render pipeline to produce enabled target files.
5. **Report.** Print each created file and a `Next:` line pointing to boundary
   declaration.

If any step after pre-flight fails, `init` rolls back files and directories it
created. `init` refuses to run when `.architect-companion/` already exists;
ongoing maintenance lives in `render` and `upgrade-profile`.

## Flags

| Flag                             | Default                                                                | Purpose                                                                                                |
| -------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `--profile <name>` (repeatable)  | auto-pick only if exactly one profile is installed                     | Select one or more profiles. Validated against installed profiles.                                     |
| `--language <name>` (repeatable) | required in non-interactive mode                                       | Values written to `project.languages`; used to activate matching implementations.                      |
| `--project-name <name>`          | derived from the project directory name (lowercased, hyphen-separated) | Value written to `project.name` in `harness.yml`.                                                      |
| `--target <key>` (repeatable)    | `agentsMd` enabled by init                                             | Enable a render target.                                                                                |
| `--no-target <key>` (repeatable) | `agentsMd` enabled by init                                             | Disable an init-default target.                                                                        |
| `--no-render`                    | render runs                                                            | Scaffold and write the profile lock, but skip the render step. Run `architect-companion render` later. |
| `--dry-run`                      | writes files                                                           | Show the plan and exit without writing.                                                                |
| `--project <dir>`                | current working directory                                              | Consuming-project root.                                                                                |
| `--profiles <dir>`               | shipped `profiles/` directory                                          | Profile discovery root.                                                                                |

Conflicting target selections, for example
`--target cursor --no-target cursor`, are rejected before any work begins.

## Behavior

### Interactive wizard vs. flag-driven

`init` runs an interactive wizard when stdin is a TTY and at least one of
project name, profiles, languages, or targets is missing from flags. In
non-interactive mode, missing profiles are auto-picked only when exactly one
profile is installed; missing languages fail with a helpful error.

Cancelling the wizard exits 130 without writing anything.

### Profile resolution

`init` reads the profiles directory and treats each subdirectory containing a
`profile.yml` as an installed profile. Repeated `--profile` flags preserve
order in `harness.yml` and in the profile lock.

### Language resolution

Languages are explicit project context. In flag-driven mode, pass at least one
`--language`. In the wizard, languages are collected as a comma-separated
prompt. Languages are open identifiers, not a closed enum.

### Target selection

Profiles do not enable or disable targets. Targets live in `harness.yml`.
`init` enables `agentsMd` by default for the first generated guidance file;
the user can add or remove targets with `--target` and `--no-target`.

### Profile lock

`init` writes `.architect-companion/profile.lock.yml` as part of the same
transaction. The lock pins every selected profile name, version, order, and
content hash.

### `--dry-run`

`--dry-run` reports what would happen and exits 0 if pre-flight passes. It
previews the bytes of `harness.yml` and `boundaries.yml`, then lists render
outputs. It does not preview rendered target content.

## Failure modes

| Condition                                             | Result                                                                      |
| ----------------------------------------------------- | --------------------------------------------------------------------------- |
| `.architect-companion/` already exists                | Refuse. Suggest `upgrade-profile` / `render` for existing harnesses.        |
| Pre-existing unmarked file at a render-target path    | Refuse with the full list of conflicting paths and a pointer to the marker. |
| `--profile <name>` not installed                      | Refuse with the missing profile name.                                       |
| No `--language` in non-interactive mode               | Refuse with a missing-language error.                                       |
| `--target` or `--no-target` references an unknown key | Refuse with the allowed key list.                                           |
| `--target` and `--no-target` name the same key        | Refuse.                                                                     |
| SIGINT during writes                                  | Roll back, exit 130, print `init cancelled.`                                |
| Render error mid-flight                               | Roll back, exit 1, print the underlying error.                              |

## Implementation

Lives in [`src/init/`](../src/init/):

- `init.ts` — orchestrator. Owns rollback, SIGINT handling, lock writing, and
  render invocation.
- `preflight.ts` — pre-flight checks: existing-directory refusal, profile
  validation, target merging, and render-target conflict detection.
- `scaffold.ts` — formatters and writers for `harness.yml` and
  `boundaries.yml`.
- `prompts.ts` — interactive wizard built on `@clack/prompts`.
- `errors.ts` — `InitError` class.
