# 0007. Server-Side Render Pipeline

## Status

Accepted

## Context

The harness model needs to be rendered into tool-specific artifacts (AGENTS.md, Cursor rules,
Claude Code skills, GitHub Actions workflows, dependency-cruiser configs). These artifacts
need to be available to all team members regardless of which local tooling they use.

An alternative is to render locally in the CLI, which is simpler but means each developer
re-renders independently and artifacts may diverge across workstations.

## Decision

Rendering is a server-side operation. The CLI sends a render request to the server, the
server loads the profile from its own storage, builds the effective harness model, runs all
renderers, stores the resulting artifacts under `data/artifacts/<projectName>/`, and returns
the files in the response.

The render API (`POST /api/render`) accepts: `profileName`, `profileVersion`, `projectName`,
`modulesYaml`, and optional `stack` and `targets` overrides. The profile YAML is loaded from
the server's profile library — the CLI does not send the profile itself.

Artifacts are keyed by project name, not by profile name or version. This is because a
single project may change profiles over time, and the current rendered state per project is
what matters.

The project name is derived at runtime from the git remote URL (`git remote get-url origin`),
not from `harness.yml`. This decouples the stored harness config from the server-side
project identity.

All renderers run on every render call regardless of targets. The `targets` field in
`harness.yml` controls which artifacts are written to disk by the CLI, not which are
generated by the server. This keeps the server artifact store complete and lets the CLI
apply selectively based on the local environment.

## Consequences

- The server must be running and must have the referenced profile before `render` can be
  called. Profile upload is a prerequisite.
- Artifacts are consistent for all team members who pull from the same server.
- Renderers are server-only code. The CLI does not contain renderer logic.
- The `data/` directory is gitignored below the top-level subdirectories; `.gitkeep` files
  track the folder structure itself.
- A future database layer can replace the filesystem store for profiles and artifacts
  without changing the render API contract.
